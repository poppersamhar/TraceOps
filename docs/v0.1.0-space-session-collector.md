# TraceOps v0.1.1 Space Evaluation Collector

## 这个版本做什么

v0.1.1 是发给数据提供同事的独立本地采集器。产品范围只覆盖 KodaX Space，但会把真实测试和企业任务 Session 完整整理成可由 v0.2.0 评测系统消费的数据包。

唯一主流程是：

```text
检测 Space Session → 一键收集 → 完整评测预处理 → 质量门禁 → 生成数据包 → 下载发送
```

它不包含评测、Harness 修改、训练集审批或模型后训练。这些能力进入独立的 v0.2.0 和 v0.3.0 产品线。

## 同事操作步骤

### 1. 准备环境

- macOS、Windows 或 Linux。
- Node.js 20 或更高版本。
- 本机已经使用过 KodaX Space，并完成至少一个真实任务。

### 2. 启动采集器

在 TraceOps 项目目录运行：

```bash
npm install
npm run collector
```

终端出现以下地址后，在浏览器打开它：

```text
http://localhost:4177/
```

### 3. 收集数据

页面会自动检测 Space Session 目录。检测成功后：

1. 点击 **收集并生成评测数据**。
2. 等待页面显示 **评测数据包已准备好**。
3. 点击 **下载评测数据包**。
4. 将下载的 `.json.gz` 文件发送给 TraceOps 负责人。

不要解压或编辑该文件；原样发送即可。

## 会读取什么

Space 与 KodaX SDK 默认共用：

```text
~/.kodax/sessions
```

如果 Space 使用了独立数据档，采集器会按以下优先级解析：

1. `KODAX_SESSIONS_DIR`
2. `KODAX_PROFILE_DIR/sessions`
3. `KODAX_HOME/sessions`
4. `~/.kodax/sessions`

采集器只读取 Session 文件，不读取浏览器 Cookie、账号密码或 Electron 缓存，也不会修改源文件。

## 自动处理规则

### 初步筛选

- 保留用户主 Session。
- 排除 `managed-task-worker` 子 Agent 内部 Session。
- 排除 `space-ephemeral` 和 `quick-ask` 临时 Session。
- 保留失败任务，因为失败经验是后续能力边界评测的重要输入。
- 没有有效对话的空 Session 不进入数据包。

### 清洗与脱敏

- 保留完整 Lineage，并标注当前有效与非有效分支，供后续分析 Agent 的回退和改写过程。
- 删除 thinking 内容。
- 替换 API Key、Token、Password、Secret 等凭据字段。
- 替换 macOS、Linux 和 Windows 本地路径。
- 替换邮箱和 localhost URL。
- 不导出原始工程路径和原始 Session ID；工具输入与结果保留结构，但会逐字段脱敏、匿名化和限长。
- 使用匿名稳定键关联 Trace、Event、Evaluation Case 与 Evidence。
- 保留有效与非有效分支、工具输入/结果和 Evidence 关系；结构化内容按字段脱敏。
- 复用 Clean Trace 计算 Evidence Coverage、压缩指标、风险和治理摘要。
- 为每个 Trace 生成 Evaluation Case、成功判据、Grader 配置和 Replay 要求。
- 检查完整性、隐私、证据、工具配对、运行归因、Benchmark 污染与包内重复。

#### 凭据风险分级（v0.1.1）

当前预处理策略为 `space-eval-preprocess-v1.1`，将凭据风险分为三级：

- `none`：未检测到凭据相关内容。
- `mention`：仅提及 `.env`、`token`、`api_key`、`password`、`secret` 等配置名、说明文本或占位符。该类数据会先脱敏，再完整保留 Trace，并以 `needs_review` 进入人工复核。
- `high_confidence`：检测到 Bearer Token、私钥块、强特征 Token 或疑似真实高熵凭据值。该类才会标记为 `privacy_blocked`，仅导出元数据和脱敏摘要。

凭据判定在脱敏前的原始文本上进行，然后再对导出内容进行递归脱敏。这样既避免遗漏真实凭据，也不会因任务中出现“请配置 API Key”等普通语句而丢失整条 Trace。

### 质量门禁

每个 Evaluation Case 都会得到一个结论：

- `eval_ready`：自动门禁通过，可在 Harness 修改前预留为 Holdout 候选。
- `needs_review`：数据有价值，但需要补 Oracle、Evidence、运行归因或企业内容审批。
- `privacy_blocked`：发现高置信真实凭据风险，只导出元数据和脱敏摘要，详细 Payload 不进入数据包。
- `incomplete`：缺少任务、最终结果或基本事件结构，不能直接用于评测。

### 输出内容

压缩数据包包含：

- `manifest`：版本、兼容契约、处理策略、质量分布、排除原因和内容校验值。
- `qualityReport`：四级门禁分布、人工复核数量、Grader 类型、Replay 模式以及失败/警告检查项聚合。
- `caseIndex`：不包含原始任务文本的轻量 Case 索引，v0.2.0 可直接据此建立 Review 队列，再通过 `sourceTraceKey` 关联完整 Trace。
- `evaluationTraces`：完整脱敏事件图、Lineage、工具输入/结果、Evidence、运行快照和预处理指标。
- `evaluationCases`：Task、Expected Outcome、Assertions、Grader、Replay 要求、能力标签和质量门禁。

所有 Case 默认是 `update_evidence`，用于发现 Agent 能力边界。只有在 Harness 修改前冻结、去重并作为独立 Holdout 保留的 Case，才能晋升为正式 Validation Benchmark。这能避免用同一批数据既改 Harness 又证明 Harness 变强。

## 数据安全说明

- 不会自动上传数据。
- 不会把数据发送到第三方服务。
- 数据包只保存在浏览器下载目录和本机服务内存中。
- 本地服务重启后，内存中的下载链接会失效，需要重新收集。
- 自动脱敏不能识别所有企业专有信息；发送前应按团队规范确认接收人和传输渠道。

## 常见问题

### 页面显示“未检测到”

先确认 KodaX Space 中已经完成过任务，并检查 `~/.kodax/sessions` 是否存在。如果 Space 使用独立数据档，请使用相同的 `KODAX_PROFILE_DIR` 或 `KODAX_HOME` 环境启动 TraceOps。

### 页面显示“暂无数据”

目录存在，但只发现临时 Session、managed worker Session 或空 Session。完成一个正常的 Space Coder / Partner 任务后重新检测。

### 页面显示“本地采集服务未连接”

页面还在，但本机采集服务没有启动或已经退出。回到 TraceOps 项目目录重新运行 `npm run collector`，再点击 **重新检测**。

### 下载链接失效

本地服务重启会清空临时数据包。重新点击 **收集并生成评测数据** 即可。

### 旧数据包中的 `privacy_blocked` 能恢复吗

不能。旧版本在生成数据包时已经不导出详细 Payload，v0.1.1 无法从包内反推。需要让数据提供者使用 v0.1.1 对原始 Space Session 重新采集。

## 版本保留策略

- `apps/collector-v0.1` 对应 v0.1.x 采集器产品线，v0.1.1 构建产物输出到 `dist/v0.1.1`。
- `apps/web` 和 `dist/v0.2.0` 对应评测平台开发线。
- v0.1.0 Tag 和 Release 继续保留；v0.1.1 作为 v0.1 产品线的当前版本。
- v0.2.0 可以复用 v0.1.x 后端处理模块，但不能覆盖 v0.1.x 前端入口、构建产物和启动命令。
