# TraceOps v0.1.x Space Workflow Collector

## 产品目标

v0.1.x 用于在数据提供者本机收集完整、可复盘、已局部脱敏的 KodaX Space Session Workflow。

它关注的不是在采集阶段判断数据是否适合评测，而是尽量忠实地保留：

- 用户提出了什么任务；
- Agent 如何拆解和推进任务；
- 使用了哪些 Skill 和工具；
- 工具输入、结果、错误和重试；
- 主 Session 与 Worker 如何协作；
- 最后得到什么结果；
- 有效分支和被回退的历史分支。

主流程是：

```text
检测 Session → 完整读取 → 字段级隐私遮罩
→ 还原 Transcript 与 Workflow → 生成索引 → 下载数据包
```

Evaluation Case、Expected Outcome、Grader、Benchmark、Holdout 和 Validation 准入不在 v0.1.x 决定，统一延后到分析与评测阶段。

## 使用方式

### 1. 准备环境

- macOS、Windows 或 Linux；
- Node.js 20 或更高版本；
- 本机已经使用过 KodaX Space。

### 2. 启动采集器

```bash
npm install
npm run collector
```

打开：

```text
http://localhost:4177/
```

### 3. 下载 Workflow 包

1. 点击 **收集完整 Workflow**；
2. 等待页面显示 **Workflow 数据包已准备好**；
3. 点击 **下载 Workflow 数据包**；
4. 将 `.json.gz` 原样发送给 TraceOps 负责人。

不要手工编辑或重新压缩数据包，否则内容校验值会失效。

### 4. 浏览本机 Session

采集器首页下方提供 **本机 Session 浏览器**：

1. 使用搜索或“主 Session / Worker”筛选找到目标记录；
2. 点击任意 Session；
3. 在统一 Workflow 时间线中查看任务与结果、阶段、Skill、消息、工具调用与结果、错误、历史分支和 Evidence；
4. 展开工具或 Evidence 条目查看脱敏后的结构化 Payload；
5. 需要排查解析问题时，再展开默认折叠的原始 Transcript 与 Event / Evidence 数据。

列表只返回脱敏标题和匿名 Session Key；完整详情在点击时按需读取，并使用与导出包相同的隐私规则。浏览操作不会修改源文件，也不会自动上传数据。

## Session 目录

采集器按以下优先级定位数据：

1. `KODAX_SESSIONS_DIR`
2. `KODAX_PROFILE_DIR/sessions`
3. `KODAX_HOME/sessions`
4. `~/.kodax/sessions`

`KODAX_PROFILE_DIR` 和 `KODAX_HOME` 需要是绝对路径。

采集器递归读取普通 `.jsonl` Session 文件，忽略隐藏文件、`.archive.jsonl` 和 `.islands.jsonl`。它不会读取浏览器 Cookie、账号密码或 Electron 缓存，也不会修改源文件。

## 保留策略

只要 Session 文件可以解析，就会进入数据包，包括：

- 普通用户 Session；
- `managed-task-worker` Session；
- 成功任务；
- 失败任务；
- `space-ephemeral` 和 `quick-ask`；
- 没有对话但仍包含可用元信息的空 Session；
- 无法确定父任务的孤立 Worker。

只有两种情况不会进入：

- 文件无法读取或不是有效 Session；
- 显式设置了 `TRACEOPS_COLLECTOR_MAX_SESSIONS`，且数量超过该上限。

默认不设置数量上限。

## 完整 Workflow 内容

每个 `sessions[]` 条目包括：

- `transcript`：所有可解析的 Message、Compaction 和 Branch Summary；
- `conversation`：当前有效分支上的可见 User、Assistant 和 System 对话；
- `events`：消息、Tool call、Tool result、Artifact、错误等事件；
- `evidence`：文件、检查结果和其他 Evidence；
- `runtime`：可安全导出的运行环境信息；
- `topology`：主 Session、Worker 和候选父子关系；
- `workflow`：Skill、工具、计划事件、执行、验证、错误和完成阶段索引；
- `overview`：首个用户任务和最终 Assistant 结果的轻量入口；
- `privacy`：本 Session 的遮罩统计。

`transcript` 是完整记录的主来源；`conversation` 和 `overview` 只是便于浏览的派生视图。

## Skill 与任务拆解

采集器从实际保存的 Session 事件中识别 Skill 使用，包括 `SKILL.md` 路径、`skill://` 引用和明确的 Skill 工具调用。没有明确证据时不会虚构 Skill 名称。

计划和任务拆解通过可观察事件保留，例如：

- Assistant 的可见计划说明；
- `update_plan`、任务列表和 Todo 相关工具事件；
- Worker 创建与交接；
- 工具调用、结果、错误、重试和验证；
- Compaction 与 Branch Summary。

私密模型 reasoning 不导出原文，但会用 `<PRIVATE_REASONING_OMITTED>` 保留该内容块所在的位置。Workflow 分析应基于可见计划、消息、工具轨迹和结果，而不是依赖隐藏推理。

## 隐私处理

隐私处理原则是：

```text
隐藏敏感值，保留 Session、字段、结构和上下文
```

例如：

```text
Authorization: Bearer abc123  → Authorization: ***
api_key: sk-...               → api_key: ***
/Users/alice/project/file.ts  → <LOCAL_PATH>/file.ts
alice@example.com             → <EMAIL>
原始 Session ID               → 稳定匿名 ID
```

即使发现高置信度 Token、API Key、密码或私钥：

- Session 仍会保留；
- Conversation 仍会保留；
- Tool call 和 Tool result 仍会保留；
- 只有具体敏感值被替换；
- `privacy.sensitiveValuesMasked` 记录处理数量。

采集器不会再生成 `privacy_blocked`，也不会把整个 Session 变成 metadata-only。

## Worker 拓扑

Worker 关系分为：

- `linked`：发现明确 Session 引用；
- `inferred`：只有一个同项目主 Session，可以推测关联；
- `ambiguous`：存在多个候选父 Session；
- `orphan`：暂时找不到父 Session。

这些状态只帮助还原 Workflow，不用于过滤或阻断数据。所有 Worker 都会完整保留。

## 数据包结构

```text
manifest
workflowReport
sessionIndex
sessions
```

- `manifest`：版本、Schema、采集策略、隐私策略、数量和 SHA-256；
- `workflowReport`：Session、对话、Skill、工具、Worker 和隐私处理汇总；
- `sessionIndex`：不展开完整 Payload 的轻量检索入口；
- `sessions`：完整脱敏 Workflow Session。

文件名格式：

```text
traceops-space-workflows-YYYYMMDD-HHMMSS-N.json.gz
```

## 与 v0.2.0 的关系

v0.2.0 可以导入新的 Workflow 包，并为每个主 Session 创建待人工建模的候选。导入不会把 Workflow 自动认定为 Evaluation Case，也不会自动授予 Validation 资格。

旧的 `traceops-space-evaluation-package-v1` 仍可由 v0.2.0 导入，避免已有数据包失效。

## 本地浏览 API

- `GET /api/v0.1/collector/sessions`：返回脱敏的 Session 轻量列表；
- `GET /api/v0.1/collector/sessions/:sessionKey`：按匿名 Key 返回完整脱敏 Workflow；
- `GET /api/v0.1/collector/status`：返回数据源和 Collector 状态。

## 数据安全

- 不会自动上传；
- 不会修改 Space 原始文件；
- 压缩包只保存在浏览器下载目录和本机服务内存；
- 服务重启后下载链接失效，需要重新收集；
- 自动遮罩不能替代企业数据授权和接收人确认；
- 数据发送前仍应遵守团队传输规范。

## 常见问题

### 页面显示“未检测到”

确认 Space 已创建过 Session，并检查 `~/.kodax/sessions`。如果使用独立 Profile，请使用相同的 `KODAX_PROFILE_DIR` 或 `KODAX_HOME` 启动 TraceOps。

### 为什么空 Session 也在包里

v0.1.2 不根据评测价值提前丢数据。空 Session 可能仍包含运行、错误或拓扑信息，是否使用由后续分析决定。

### 为什么看不到完整 thinking

私密模型 reasoning 不作为可导出 Workflow 内容。包中保留其位置标记，并完整保留可见计划、工具调用、结果、错误、分支摘要和最终回答。

### 下载链接失效

本地服务重启会清空临时数据包。重新点击 **收集完整 Workflow** 即可。

## 版本保留策略

- `apps/collector-v0.1` 和 `dist/v0.1.2` 对应当前 Workflow Collector；
- v0.1.0 与 v0.1.1 Tag 继续保留供回滚；
- 后续产品可以消费 v0.1.x 数据，但不能覆盖 v0.1.x 的入口、启动命令和构建产物。
